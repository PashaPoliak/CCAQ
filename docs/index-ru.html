Клод API — Основы

1.1 Структура запроса API

API Claude использует модель «запрос-ответ». Каждый запрос к API сообщений Claude включает следующие ключевые поля: model — выбор модели (claude-opus-4-6, claude-sonnet-4-6, claude-haiku-4-5); max_tokens — максимальное количество токенов в ответе ; system — системная подсказка (определяет поведение модели); messages — история диалога (необходимо отправить полную историю для сохранения согласованности); tools — определения доступных инструментов; tool_choice — стратегия выбора инструмента. 
{
  "model": "claude-sonnet-4-6",
  "max_tokens": 1024,
  "system": "You are a helpful assistant.",
  "messages": [
    {"role": "user", "content": "Hi!"},
    {"role": "assistant", "content": "Hello!"},
    {"role": "user", "content": "How are you?"}
  ],
  "tools": [...],
  "tool_choice": {"type": "auto"}
}

1.2 Роли сообщений

Массив messages использует три роли: user — сообщения пользователя; assistant — ответы модели (включаются при отправке истории); tool — результаты вызова инструмента (отображаются как блок содержимого tool_result).
Крайне важно: при каждом запросе к API необходимо отправлять полную историю переписки . Модель не сохраняет состояние между запросами — каждый вызов независим.

1.3 Поле stop_reason

Ответ API Claude включает в себя stop_reasonинформацию о причине прекращения генерации моделью данных. Для агентных систем наиболее важными являются параметры tool_use и end_turn — они управляют циклом работы агента.
Ценить Описание Действие
"end_turn" Модель завершила свой ответ. Показать результат пользователю
"использование_инструмента" Модель хочет вызвать инструмент. Запустите инструмент и верните результат.
"max_tokens" Достигнут лимит токенов Ответ усечен; увеличьте предел.
"stop_sequence" Была обнаружена стоп-последовательность. Обрабатывайте в соответствии с логикой вашего приложения.

1.4 Системная подсказка

Системная подсказка — это специальная инструкция, определяющая контекст и правила поведения. Она: • Не является частью массива сообщений; передается отдельно в соответствующем systemполе • Имеет приоритет над сообщениями пользователя • Загружается один раз и применяется на протяжении всего диалога • Используется для определения роли, ограничений и формата вывода
Важно для экзамена: формулировка системных подсказок может привести к непреднамеренным ассоциациям инструментов. Например, инструкция типа «всегда проверяйте клиента» может привести к чрезмерному использованию модели get_customer, даже если это не требуется.

1.5 Контекстное окно

Окно контекста — это общий объем текста (в токенах), который модель может обработать за один раз. Оно включает в себя: системную подсказку, полную историю сообщений, определения инструментов и результаты работы инструментов. Ключевые проблемы окна контекста: 1. Эффект «потери посередине»: модели надежно обрабатывают информацию в начале и конце длинного входного потока, но могут упускать детали в середине. Решение: размещайте ключевую информацию ближе к началу или концу. 2. Накопление результатов работы инструментов: каждый вызов инструмента добавляет выходные данные в контекст. Если инструмент возвращает более 40 полей, но важны только 5, большая часть контекста теряется. 3. Прогрессивное суммирование: при сжатии истории числовые значения, проценты и даты часто теряются и становятся нечеткими («примерно», «приблизительно», «несколько»).

Инструменты и использование инструментов

2.1 Что такое tool_use?

tool_useЭто механизм, позволяющий Клоду вызывать внешние функции. Модель не выполняет код напрямую — она генерирует структурированный запрос на вызов инструмента; ваш код выполняет его и возвращает результат. Последовательность действий: 1. Вы определяете инструменты в запросе API. 2. Клод решает вызвать инструмент и возвращает блок кода. tool_use3. Ваш код выполняет инструмент. 4. Вы возвращаете результат в виде tool_resultблока кода. 5. Клод продолжает выполнение на основе полученного результата.

2.2 Определение инструмента

Каждый инструмент определяется с помощью JSON-схемы. Критически важные аспекты описания инструмента: 1. Описание является основным механизмом выбора . Модель LLM выбирает инструменты на основе их описаний. Минимальные описания приводят к ошибкам, когда инструменты дублируются. 2. Включите в описание: что делает и возвращает инструмент, форматы входных данных и примеры значений, граничные случаи и ограничения, когда следует использовать этот инструмент по сравнению с аналогичными альтернативами. 3. Избегайте идентичных или дублирующих описаний для разных инструментов. Если analyze_contentи analyze_documentимеют почти идентичные описания, модель будет их путать. 4. Встроенные инструменты против инструментов MCP: агенты могут предпочитать встроенные инструменты (Read, Grep) инструментам MCP с аналогичной функциональностью. Усильте описания инструментов MCP, чтобы подчеркнуть конкретные преимущества. 
{
  "name": "get_customer",
  "description": "Finds a customer by email or ID. Returns the customer
  profile, including name, email, order history, and account status.
  Use this tool BEFORE lookup_order to verify the customer's identity.
  Accepts an email (format: user@domain.com) or a numeric customer_id.",
  "input_schema": {
    "type": "object",
    "properties": {
      "email": {
        "type": "string",
        "description": "Customer email"
      },
      "customer_id": {
        "type": "integer",
        "description": "Numeric customer ID"
      }
    },
    "required": []
  }
}

2.3 Параметр tool_choice

tool_choiceуправляет тем, как модель выбирает инструменты. Важные сценарии: • tool_choice: "any"+ несколько инструментов извлечения → модель выбирает лучший, но вы все равно получаете структурированный результат • Принудительный выбор → когда необходимо гарантировать выполнение определенного первого действия (например, extract_metadataперед обогащением)
Ценить Поведение Когда использовать
{"type": "auto"} Модель решает, следует ли вызвать инструмент или ответить текстом. В большинстве случаев используется значение по умолчанию.
{"type": "any"} Модель должна вызвать какой-либо инструмент. Когда вам необходим гарантированный структурированный результат
{"type": "tool", "name": "extract_metadata"} Модель должна вызывать конкретный инструмент. Когда необходим вынужденный первый шаг

2.4 JSON-схемы для структурированного вывода

Использование tool_useJSON-схем — наиболее надежный способ получения структурированного вывода из Claude. Он: • Гарантирует синтаксически корректный JSON (отсутствие пропущенных фигурных скобок, замыкающих запятых) • Обеспечивает требуемую структуру (присутствие обязательных полей) • Не гарантирует семантическую корректность (значения все еще могут быть неверными) Правила проектирования схемы: 1. Обязательные и необязательные: помечайте поля как обязательные только в том случае, если информация всегда доступна. Обязательные поля заставляют модель создавать искусственные значения при отсутствии данных. 2. Поля, допускающие значение NULL: используйте "type": ["string", "null"]для информации, которая может отсутствовать. Модель может вернуть NULL вместо искусственного создания значений. 3. Перечисления с "other": добавьте "other" + строку с подробной информацией, чтобы избежать потери данных за пределами ваших предопределенных категорий. 4. Перечисление "unclear": для случаев, когда модель не может с уверенностью выбрать категорию — честное "unclear" лучше, чем неправильная категория. 
{
  "type": "object",
  "properties": {
    "category": {
      "type": "string",
      "enum": ["bug", "feature", "docs", "unclear", "other"]
    },
    "category_detail": {
      "type": ["string", "null"],
      "description": "Details if category = 'other' or 'unclear'"
    },
    "severity": {
      "type": "string",
      "enum": ["critical", "high", "medium", "low"]
    },
    "confidence": {
      "type": "number",
      "minimum": 0,
      "maximum": 1
    }
  },
  "required": ["category", "severity"]
}

2.5 Синтаксические и семантические ошибки

Понимание различий между типами ошибок имеет решающее значение для проектирования надежных систем.
Тип ошибки Пример Смягчение последствий
Синтаксис Неверный JSON, неправильный тип поля. Использование инструмента с JSON-схемой (исключает)
Семантический Итоговые суммы не сходятся, значение указано не в том поле, галлюцинация. Проверка достоверности данных, повторная попытка с обратной связью, самокоррекция.

Claude Agent SDK — Agentic Systems

3.1 Агентный цикл

Основной шаблон автономного выполнения задач — это агентный цикл: 1. Отправка запроса Клоду с инструментами. 2. Получение ответа. 3. Проверка stop_reason: • "tool_use" → запуск инструмента, добавление результата в историю, возврат к шагу 1. • "end_turn" → задача выполнена, отображение результата пользователю. 4. Повторение до завершения. Антипаттерны (избегать): • Анализ текста помощника для определения завершения ("Задача выполнена") • Использование произвольного ограничения на количество итераций (например, max_iterations=5) в качестве основного условия остановки • Проверка того, выдал ли помощник текстовое содержимое в качестве сигнала о завершении
Правильный подход: единственным надежным сигналом завершения является stop_reason == "end_turn".

3.2 Конфигурация AgentDefinition

AgentDefinition — это объект конфигурации агента в SDK Claude Agent. Ключевые параметры: • name / description— идентификация и описание агента • system_prompt— системная подсказка с инструкциями • allowed_tools— список разрешенных инструментов (принцип минимальных привилегий) 
agent = AgentDefinition(
    name="customer_support",
    description="Handles customer requests for returns and order issues",
    system_prompt="You are a customer support agent...",
    allowed_tools=[
        "get_customer",
        "lookup_order",
        "process_refund",
        "escalate_to_human"
    ]
    # For a coordinator:
    # allowed_tools=["Task", "get_customer", ...]
)

3.3 Архитектура "центр-периферия"

Многоагентная архитектура обычно строится по принципу «звезда», где координатор находится в центре. Координатор отвечает за: • Разложение задачи на подзадачи • Определение необходимых субагентов (динамический выбор) • Делегирование работы субагентам • Агрегирование и проверку результатов • Обработку ошибок и повторных попыток • Передачу результатов пользователю
Ключевой принцип: субагенты имеют изолированный контекст. • Субагенты не наследуют автоматически историю разговоров координатора. • Весь необходимый контекст должен быть явно передан в подсказке субагента. • Субагенты не используют общую память для разных вызовов. • Все коммуникации проходят через координатора (для обеспечения наблюдаемости и контроля ошибок). 
# Hub-and-Spoke topology
       Coordinator
       /    |    \
      /     |     \
Subagent1 Subagent2 Subagent3
(search)  (analysis) (synthesis)

3.4 Инструмент для создания подач агентов

Вспомогательные агенты создаются с помощью инструмента «Задачи». Координатор allowedToolsдолжен включать в себя "Task". Явная передача контекста обязательна. Координатор может вызывать несколько задач в одном ответе — вспомогательные агенты выполняются параллельно. 
# Bad: the subagent has no context
Task: "Analyze the document"

# Good: full context in the prompt
Task: "Analyze the following document.
Document: [full document text]
Prior search results: [web search results]
Output format requirements: [schema]"

# Parallel spawning in one coordinator response:
Task 1: "Search for articles about X"
Task 2: "Analyze document Y"
Task 3: "Search for articles about Z"
# All three run concurrently

3.5 Перехватчики в SDK агента

«Перехватчики» позволяют перехватывать и преобразовывать данные в определенных точках жизненного цикла агента. Ключевое различие: «перехватчики» против подсказок.
Атрибут Крючки Подсказка по использованию
Гарантия Детерминированный (100%) Вероятностный (>90%, а не 100%)
Когда использовать Ключевые бизнес-правила, финансовые операции, соответствие нормативным требованиям. Общие предпочтения, рекомендации, форматирование
Пример Возврат средств при блокировке > 500 долларов «Постарайтесь решить проблему, прежде чем обострять ситуацию». 
# PostToolUse: normalize date formats
@hook("PostToolUse")
def normalize_dates(tool_result):
    # Convert Unix timestamp -> ISO 8601
    return normalized_result

# PreToolUse: enforce business rules
@hook("PreToolUse")
def enforce_refund_limit(tool_call):
    if tool_call.name == "process_refund" and tool_call.args.amount > 500:
        return redirect_to_escalation(tool_call)

Протокол контекста модели (MCP)

4.1 Что такое MCP?

Протокол контекста модели (MCP) — это открытый протокол для подключения внешних систем к Claude. MCP определяет три основных типа ресурсов: 1. Инструменты — функции, которые агент может вызывать для выполнения действий (операции CRUD, вызовы API, выполнение команд); 2. Ресурсы — данные, которые агент может считывать для контекста (документация, схемы баз данных, каталоги контента); 3. Подсказки — предопределенные шаблоны подсказок для распространенных задач.

4.2 Серверы MCP

Сервер MCP — это процесс, реализующий протокол MCP и предоставляющий инструменты/ресурсы. При подключении к серверу MCP: • Все инструменты обнаруживаются автоматически • Инструменты со всех подключенных серверов становятся доступны одновременно • Описания инструментов определяют, как модель будет их использовать. Выбор серверов: • Для стандартных интеграций (Jira, GitHub, Slack) отдавайте предпочтение существующим серверам MCP, созданным сообществом • Создавайте собственные серверы только для уникальных рабочих процессов, специфичных для конкретной команды

4.3 Настройка серверов MCP

Конфигурация проекта (.mcp.json) — для командного использования: • Хранится в корне проекта и управляется системой контроля версий • Переменные окружения ( ${GITHUB_TOKEN}) используются для секретов — сами токены не добавляются в репозиторий • Доступна всем участникам проекта Конфигурация пользователя (~/.claude.json) — для личных/экспериментальных серверов: • Хранится в домашнем каталоге пользователя • Не передается через систему контроля версий • Подходит для личных экспериментов и тестирования 
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "jira": {
      "command": "npx",
      "args": ["-y", "mcp-server-jira"],
      "env": {
        "JIRA_TOKEN": "${JIRA_TOKEN}"
      }
    }
  }
}

4.4 Флаг isError в MCP

Когда инструмент MCP обнаруживает ошибку, он использует её isError: trueв ответе. Обычная ошибка не предоставляет агенту никакой информации для принятия решения. 
// Structured error (good):
{
  "isError": true,
  "content": {
    "errorCategory": "transient",
    "isRetryable": true,
    "message": "Temporary timeout while calling the orders API.",
    "attempted_query": "order_id=12345",
    "partial_results": null
  }
}

// Generic error (anti-pattern):
{
  "isError": true,
  "content": "Operation failed"
}

4.5 Ресурсы MCP

Ресурсы — это данные, которые агент может запросить для получения контекста без совершения каких-либо действий: • Каталоги контента (например, список всех задач проекта, иерархическая навигация) • Схемы баз данных (понимание структуры данных) • Документация (справочники API, внутренние руководства) • Сводки по задачам/проблемам
Преимущество ресурса: агенту не требуются запросы к инструментам исследования, чтобы понять, какие данные существуют. Ресурс предоставляет мгновенную «карту».

Код Клода — Конфигурация

5.1 Иерархия CLAUDE.md

CLAUDE.md — это файл инструкций для Claude Code. Существует трехуровневая иерархия: 1. Уровень пользователя: ~/.claude/CLAUDE.md • Применяется только к данному пользователю • НЕ распространяется через систему контроля версий • Личные предпочтения и стиль работы 2. Уровень проекта: .claude/CLAUDE.md или корневой каталог CLAUDE.md • Применяется ко всем участникам проекта • Управляется через систему контроля версий • Стандарты кодирования, стандарты тестирования, архитектурные решения 3. Уровень каталога: CLAUDE.md в подкаталогах • Применяется при работе с файлами в этом каталоге • Соглашения, специфичные для этой части кодовой базы
Распространенная ошибка: новый член команды не получает инструкции по проекту, потому что они были размещены на ~/.claude/CLAUDE.mdуровне пользователя, а не на .claude/CLAUDE.mdуровне проекта.

5.2 Синтаксис @path (импорт файлов)

Файл CLAUDE.md может ссылаться на внешние файлы с помощью `@path` @path, что делает конфигурацию модульной. Правила для `@path`: • Используйте @`@path` непосредственно перед путем к файлу (без пробела) • Поддерживаются относительные и абсолютные пути • Относительные пути определяются относительно файла, содержащего импорт • Максимальная глубина вложенности импорта — 5

5.3 Каталог .claude/rules/

.claude/rules/Это альтернатива монолитному файлу CLAUDE.md, используемая для организации правил по темам. Ключевая особенность: метаданные в формате YAML с путями для условной загрузки — правило загружается только тогда, когда Claude Code редактирует файл, соответствующий шаблону путей. Когда использовать: • .claude/rules/с путями — когда соглашения применяются к файлам, разбросанным по многим каталогам (тесты, миграции) • CLAUDE.md на уровне каталога — когда соглашения привязаны к конкретному каталогу 
# .claude/rules/api-conventions.md
---
paths: ["src/api/**/*"]
---
For API files, use async/await with explicit error handling.
Each endpoint must return a standard response wrapper.

# .claude/rules/testing.md
---
paths: ["**/*.test.tsx", "**/*.test.ts"]
---
Tests must use describe/it blocks.
Use data factories instead of hardcoding.
Do not mock the database—use a test database.

5.4 Пользовательские команды и навыки с использованием слэша

Навыки представляют собой многократно используемые шаблоны подсказок, вызываемые через /name. Команды проекта ( .claude/commands/или .claude/skills/): • Хранятся в системе контроля версий и доступны всем при клонировании репозитория • Обеспечивают согласованность рабочих процессов в команде Команды пользователей ( ~/.claude/commands/или ~/.claude/skills/): • Персональные команды, не передаваемые через систему контроля версий • Для индивидуальных рабочих процессов Параметры метаданных:
Параметр Описание

5.5 Режим планирования против прямого исполнения

Режим планирования: • Модель только исследует и планирует; она не вносит изменений • Использует Read, Grep, Glob для исследования кодовой базы • Создает план реализации, который утверждается пользователем • Безопасное исследование без побочных эффектов Когда использовать режим планирования: • Большие изменения (десятки файлов) • Несколько возможных подходов • Архитектурные решения • Незнакомая кодовая база • Миграции библиотек, затрагивающие более 45 файлов Когда использовать прямое выполнение: • Исправление одного файла с четким трассировочным стеком • Добавление одной проверки валидации • Хорошо понятные, однозначные изменения
Подпрограмма Explore — специализированный подпрограмма для исследования кодовой базы: изолирует подробный вывод от основного контекста, возвращает только краткое описание, предотвращает исчерпание контекстного окна в многоэтапных задачах.

5.6 Команды /compact и /memory

/compact — сжатие контекста: • Обобщает предыдущую историю, чтобы освободить контекстное окно • Используется в длительных сеансах исследования, когда контекст заполняется подробным выводом инструмента • Риск: точные числовые значения, даты и конкретные детали могут быть потеряны во время обобщения /memory — управление памятью между сеансами: • Открывает файл CLAUDE.md для редактирования • Информация сохраняется между сеансами и автоматически загружается при запуске • Полезно для хранения соглашений проекта, пользовательских настроек, часто используемых команд и текущего рабочего контекста

5.7 Claude Code CLI для CI/CD

Флаг -p(или --print): Неинтерактивный режим — обрабатывает приглашение, выводит в стандартный поток вывода, завершает работу. Единственный правильный способ запуска Claude в конвейерах CI/CD. Предотвращение дублирования комментариев: При повторном просмотре после новых коммитов включайте результаты предыдущего просмотра в контекст и дайте указание Claude сообщать только о новых или нерешенных проблемах. 
# Non-interactive CI mode
claude -p "Analyze this pull request for security issues"

# Structured output for CI
claude -p "Review this PR" \
  --output-format json \
  --json-schema '{"type":"object",...}'

6.1 Подсказка с малым количеством попыток

Использование примеров с малым количеством примеров (few-shot prompting) подразумевает включение 2–4 примеров ввода/вывода в запрос для демонстрации ожидаемого поведения. Почему использование примеров с малым количеством примеров эффективнее текстовых описаний: • Нечеткая инструкция, например, «будьте точнее», может быть истолкована по-разному. • Пример однозначно демонстрирует ожидаемый формат и логику принятия решений. • Модель обобщает шаблон на новые случаи. Типы примеров с малым количеством примеров: 1. Неоднозначные сценарии — демонстрация правильного выбора инструмента с обоснованием. 2. Форматирование вывода — демонстрация точно требуемого формата. 3. Допустимый и проблемный код — демонстрация того, что следует отметить, а что игнорировать. 4. Извлечение данных из документов разных форматов — обработка вариаций формата. 
# Example: ambiguous scenarios with rationale
Request: "My order is broken"
Action: Call get_customer -> lookup_order -> check status.
Rationale: "broken" may mean a damaged item; need order details.

Request: "Get me a manager"

Rationale: Customer explicitly requests a human. Do not attempt to solve.

6.2 Четкие критерии против расплывчатых инструкций

Четко сформулированные критерии более эффективны, чем расплывчатые инструкции. 
# Bad (vague):
"Check code comments for accuracy."
"Be conservative—report only high-confidence findings."

# Good (explicit criteria):
Flag a comment as problematic ONLY if:
1. The comment describes behavior that CONTRADICTS the actual code behavior
2. The comment references a non-existent function or variable
3. A TODO/FIXME comment refers to a bug that has already been fixed

Do NOT flag:
- Comments that are merely stylistically outdated
- Comments with minor wording inaccuracies
- Missing comments (that is a separate category)

# Severity criteria with examples:
CRITICAL: Runtime failure for users
  Example: NullPointerException while processing a payment
HIGH: Security vulnerability
  Example: SQL injection, XSS, missing authorization checks
MEDIUM: Logic bug without immediate impact
  Example: Wrong sorting, off-by-one error
LOW: Code quality
  Example: Duplication, suboptimal algorithm

6.3 Последовательность подсказок

Метод цепочек подсказок разбивает сложную задачу на последовательность целенаправленных шагов. Почему это важно: • Позволяет избежать рассеивания внимания — когда модель получает слишком много файлов одновременно, она может пропустить ошибки в некоторых файлах • Обеспечивает стабильное качество анализа для каждого файла • Позволяет проводить отдельный анализ взаимодействий между файлами. Когда использовать метод цепочек подсказок, а когда динамическую декомпозицию: • Метод цепочек подсказок — предсказуемые, повторяющиеся задачи (проверка кода, миграция файлов) • Динамическая декомпозиция — открытые исследования, где подзадачи становятся ясными только во время выполнения 
# Multi-pass code review
Step 1: Analyze auth.ts (local issues only)
-> Output: list of issues in auth.ts

Step 2: Analyze database.ts (local issues only)
-> Output: list of issues in database.ts

Step 3: Integration pass (cross-file dependencies)
-> Output: issues at module boundaries

6.4 Структура интервью

Перед внедрением решения Клод задает уточняющие вопросы. Это полезно в следующих случаях: • Если предметная область незнакома (финтех, здравоохранение, юридические системы) • Если задачи имеют неочевидные последствия (стратегии кэширования, режимы отказов) • Если существует несколько жизнеспособных подходов, и лучший выбор зависит от контекста 
Claude: "Before implementing caching for the API, a few questions:
1. Which cache invalidation strategy do you prefer—TTL or event-based?
2. Is stale data acceptable when the cache is unavailable?
3. Should caching be per-user or global?
4. What is the expected data volume to cache?"

6.5 Проверка и повторная попытка с обратной связью

Если извлеченные данные не проходят проверку, повторите попытку, указав контекст, включающий исходный документ, предыдущую (неправильную) попытку извлечения и конкретную ошибку. Повторная попытка будет эффективна в следующих случаях: • Ошибки форматирования (дата в неправильном формате) • Структурные ошибки (поле расположено не в том месте) • Арифметические несоответствия (модель может перепроверить данные). Повторная попытка НЕ ​​поможет в следующих случаях: • Информация отсутствует в исходном документе • Необходимый контекст находится в другом документе и не предоставлен 
# Retry with error feedback
Step 1: Extract data from the document
Step 2: Validate (Pydantic, JSON Schema, business rules)
Step 3: If error—retry with context:
  {
    "original_document": "...",
    "previous_extraction": { "total": 150, "line_items": [75, 70] },
    "error": "Field 'total' = 150, but sum(line_items) = 145. Re-check."
  }

API для пакетной обработки сообщений

7.1 Обзор

API для пакетной обработки сообщений позволяет отправлять пакеты запросов для асинхронной обработки.
Атрибут Ценить
Экономия 50% по сравнению с синхронными вызовами
Окно обработки До 24 часов (гарантия отсутствия задержек в рамках соглашения об уровне обслуживания)
Многооборотный инструмент Не поддерживается (один запрос = один ответ)
Корреляция Поле custom_id для связи запроса и ответа.

7.2 Пакетный и синхронный API

Выбирайте подходящий API в зависимости от того, является ли задача блокирующей или неблокирующей.
Задача API Почему
Предварительная проверка PR перед слиянием Синхронный Разработчик ждет; 24 часа — это неприемлемо.
Отчет о технологическом долге за ночь Партия Результат должен быть получен к утру; экономия 50%.
Еженедельный аудит безопасности Партия Не срочно; экономия 50%.
Интерактивный анализ кода Синхронный Требуется немедленный ответ.
Обработка 10 000 документов. Партия Массовая обработка; значительная экономия.

7.3 Использование custom_id

custom_idЭто позволяет: • Связать результат с исходным документом • В случае неудачи повторно отправить только документы, не прошедшие проверку • Избежать повторной обработки успешно прошедших проверку документов 
{
  "custom_id": "doc-invoice-2024-001",
  "params": {
    "model": "claude-sonnet-4-6",
    "max_tokens": 1024,
    "messages": [
      {"role": "user", "content": "Extract data from: ..."}
    ]
  }
}

7.4 Обработка сбоев в пакетной обработке

Схема обработки ошибок: 1. Отправка пакета из 100 документов. 2. 95 успешно обработаны; 5 не обработаны (превышен лимит контекста). 3. Выявление ошибок. custom_id 4. Изменение стратегии (например, разделение длинных документов на части). 5. Повторная отправка только 5 документов, не обработанных.

7.5 Планирование SLA

Если вам нужен результат через 30 часов, а обработка через Batch API может занять до 24 часов: • Период отправки: 30 - 24 = 6 часов • Пакеты должны быть отправлены не позднее чем за 24 часа до крайнего срока • Для частых отправок разделите на 4-часовые интервалы
Пакетный API обеспечивает 50% экономии средств, но не гарантирует низкую задержку . Используйте его только для неблокирующих рабочих нагрузок.

Декомпозиция задач

8.1 Фиксированные конвейеры (быстрая цепочка)

Каждый шаг определен заранее. Структура задачи предсказуема. Когда использовать: • Структура задачи предсказуема (проверки всегда следуют одному и тому же шаблону) • Все шаги известны заранее • Вам необходима стабильность и воспроизводимость 
# Fixed pipeline for document processing
Document
  → Metadata extraction
  → Data extraction
  → Validation
  → Enrichment
  → Final output

8.2 Динамическое адаптивное разложение

Подзадачи генерируются на основе промежуточных результатов. Когда использовать: • Открытые исследовательские задачи • Когда полный объем неизвестен заранее • Когда каждый шаг зависит от результатов предыдущего шага 
# Dynamic decomposition example
"Add tests for a legacy codebase"
→ First: map the structure (Glob, Grep)
→ Found: 3 modules with no tests, 2 with partial coverage
→ Prioritize: start with the payments module (high risk)
→ During work: discovered a dependency on an external API
→ Adapt: add a mock for the external API before writing tests

8.3 Многопроходный анализ кода

Для запросов на слияние, содержащих более 10 файлов, используйте многопроходный подход. Почему один проход по 14 файлам плох: • Рассредоточение внимания: глубокий анализ одних файлов, поверхностный для других • Несогласованные комментарии: шаблон отмечен в одном файле, но одобрен в другом • Пропущенные ошибки: очевидные ошибки пропускаются из-за когнитивной перегрузки 
# Multi-pass approach
Pass 1 (per-file):
- Analyze auth.ts → list local issues
- Analyze database.ts → list local issues
- Analyze routes.ts → list local issues

Pass 2 (integration):
- Analyze relationships between files
→ Cross-file issues: inconsistent types, circular dependencies

Эскалация и участие человека в процессе

9.1 Когда следует обратиться к человеку для эскалации проблемы

Надежные триггеры эскалации:
Ситуация Действие
Клиент прямо просит позвать менеджера. Немедленно обратитесь к вышестоящему руководству; не пытайтесь решить проблему самостоятельно.
Данная политика не распространяется на этот запрос. Применить эскалацию (например, сравнение цен с конкурентами, если в политике компании об этом ничего не говорится).
Агент не может продвинуться вперед После разумного количества попыток следует перейти на более высокий уровень.
Финансовая операция, превышающая пороговое значение Применить эскалацию (желательно с помощью обработчика событий, а не подсказки).
Множественные совпадения при поиске клиента Запросите дополнительные идентификаторы; не гадайте.
Ненадежные методы эскалации:
Ненадежный метод Почему это терпит неудачу
Анализ настроений Настроение клиента не коррелирует со сложностью дела.
Модель самооценки уверенности (1–10) Модель может быть заведомо неверной; калибровка некачественная.
Автоматический классификатор Излишняя сложность; может потребоваться обучающие данные, которых у вас нет.

9.2 Схемы эскалации

Немедленная эскалация (явный запрос): 
Customer: "I want to speak to a manager"
Agent: [immediately calls escalate_to_human]
# NOT: "I can help with your issue, let me..."

# Nuanced escalation pattern:
# acknowledge → resolve → escalate on reiteration
Customer: "This is outrageous, I'm very unhappy!"
Agent: [acknowledges] "I understand your frustration."
Agent: [offers resolution] "I can offer a replacement or a refund."
Customer: "No, I want to talk to someone!"
Agent: [customer insists again → immediate escalation]

9.3 Структурированные протоколы передачи управления

При эскалации проблемы оператор должен передать структурированное резюме человеку. У оператора нет доступа к полной стенограмме разговора — резюме должно быть полным и самодостаточным. 
{
  "customer_id": "CUST-12345",
  "customer_name": "John Smith",
  "issue_summary": "Refund request for a damaged item",
  "order_id": "ORD-67890",
  "root_cause": "Item arrived damaged; photos attached",
  "actions_taken": [
    "Verified customer via get_customer",
    "Confirmed order via lookup_order",
    "Offered a standard replacement — customer insists on a refund"
  ],
  "refund_amount": "$89.99",
  "recommended_action": "Approve a full refund",
  "escalation_reason": "Customer requested to speak with a manager"
}

9.4 Калибровка достоверности

Для систем извлечения данных: 1. Оценка достоверности на уровне полей: модель выдает оценку достоверности для каждого извлеченного поля. 2. Калибровка: используйте размеченные наборы данных для проверки, чтобы настроить пороговые значения. 3. Маршрутизация: • Высокая достоверность + стабильная точность → автоматическая обработка • Низкая достоверность или неоднозначные источники → проверка человеком
Стратифицированная случайная выборка: даже при высокой степени достоверности выборки регулярно проверяйте ее. Совокупная точность в 97% может скрывать 40% ошибок для конкретного типа документа. Анализируйте точность по типу документа и по полям, а не только в целом.

Обработка ошибок в многоагентных системах

10.1 Категории ошибок

Понимание категорий ошибок определяет правильную стратегию восстановления.
Категория Примеры Повторная попытка Действия агента
Переходный Истекло время ожидания, ошибка 503, сбой сети. Да Повторить попытку с экспоненциальной задержкой
Проверка Неверный формат ввода, отсутствует обязательное поле. Нет Измените запрос и повторите попытку.
Бизнес Нарушение правил, превышен пороговый уровень Нет Объясните пользователю; предложите альтернативу
Разрешение Доступ запрещен Нет Усиление

10.2 Антипаттерны обработки ошибок

Распространенные антипаттерны, которых следует избегать:
Антипаттерн Проблема Правильный подход
Общий статус "поиск недоступен" Координатор не может решить, как восстановиться. Возвращать тип ошибки, запрос, частичные результаты, альтернативы.
Молчаливое подавление (пустой результат = успех) Координатор считает, что матчей не было, но это был провал. Различайте «нет результатов» и «неудачный поиск».
При возникновении одной ошибки весь рабочий процесс прерывается. Вы теряете все частичные результаты. Продолжить обработку частичных результатов; отметить пробелы.
Бесконечные попытки внутри субагента Задержка и нерациональное использование ресурсов Локальное восстановление (1–2 повторные попытки), затем распространение.

10.3 Структурированная ошибка субагента

Хорошо структурированная ошибка предоставляет координатору всю необходимую информацию для принятия решения о восстановлении. 
{
  "status": "partial_failure",
  "failure_type": "timeout",
  "attempted_query": "AI impact on music industry 2024",
  "partial_results": [
    {
      "title": "AI Music Generation Report",
      "url": "...",
      "relevance": 0.8
    }
  ],
  "alternative_approaches": [
    "Try a narrower query: 'AI music composition tools'",
    "Use an alternative data source"
  ],
  "coverage_impact": "Not covered: AI impact on music production"
}

Управление контекстом в производстве

11.1 Блок «Факты дела, сохраняющиеся в силе»

Вместо того чтобы полагаться на историю разговора (которая ухудшается при составлении резюме), извлеките ключевые факты в структурированный блок, который будет включен в каждый вопрос. 
=== CASE FACTS (updated whenever a new fact appears) ===
Customer ID: CUST-12345
Order ID: ORD-67890
Order Date: 2025-01-15
Order Amount: $89.99
Issue: Damaged item on delivery
Customer Request: Full refund
Status: Pending manager approval
===

11.2 Результаты работы инструмента для обрезки

Если lookup_orderвозвращается более 40 полей, а для текущей задачи вам нужно только 5, используйте хук PostToolUse для обрезки вывода. Это сохранит контекст и уменьшит лишнюю информацию. 
# PostToolUse hook: keep only relevant fields
@hook("PostToolUse", tool="lookup_order")
def trim_order_fields(result):
    return {
        "order_id": result["order_id"],
        "status": result["status"],
        "total": result["total"],
        "items": result["items"],
        "return_eligible": result["return_eligible"]
    }

11.3 Ввод с учетом положения (потеря в середине)

При размещении важной информации учитывайте эффект потери деталей в середине. Модели надежно обрабатывают начало и конец длинных входных данных, но могут упускать детали в середине. 
# Structure to mitigate lost-in-the-middle
[KEY FINDINGS — at the top]
Found 3 critical vulnerabilities...

[DETAILED RESULTS — middle]
=== File auth.ts ===
...

[ACTION ITEMS — at the end]
Priority: fix auth.ts vulnerabilities before merge.

11.4 Делегирование полномочий субагентам для защиты контекста

Используйте субагентов для изоляции подробной работы от основного контекста. Основной агент: "Исследовать зависимости модуля платежей" → Субагент (Explore): читает 15 файлов, отслеживает импорты → Возвращает: "Модуль платежей зависит от AuthService, OrderModel и внешнего API PaymentGateway" → Основной агент: сохраняет одну строку в контексте вместо 15 файлов
Ограниченные бюджеты контекста: отправляйте субагентам минимальный контекст: конкретную задачу + необходимые данные. Дайте субагентам указание возвращать структурированные результаты, а не необработанные данные. Используйте allowedTools, чтобы ограничить набор инструментов субагента — меньше инструментов означает меньше отвлекающих факторов и меньшие затраты контекста.

Советы по подготовке к экзамену — Области применения и сценарии

5 областей экзамена

Сертификационный экзамен охватывает 5 областей с разным весом:
Домен Масса Ключевые темы
1. Архитектура и оркестровка агентов 27% Агентная модель, модель «звезда», инструмент для работы с задачами, хуки, управление сессиями, декомпозиция задач.
2. Разработка инструментария и интеграция MCP 18% Описание инструментов, isError, выбор инструмента, конфигурация сервера MCP, встроенные инструменты
3. Настройка кода и рабочие процессы Claude 20% Иерархия CLAUDE.md, @path, .claude/rules/, навыки, режим планирования, CI/CD
4. Оперативное проектирование и структурированный результат. 20% Ограниченное количество примеров, четкие критерии, цепочка подсказок, JSON-схемы, проверка/повторная попытка, пакетный API
5. Управление контекстом и надежность 15% Потерянные посередине, факты дела, распространение ошибок, эскалация, калибровка достоверности

Область 1: Ключевые понятия архитектуры агентов

Цикл работы агента: stop_reason = "end_turn" — ЕДИНСТВЕННЫЙ надежный сигнал остановки. Никогда не анализируйте текст и не используйте количество итераций в качестве основного условия остановки. Центр-периферия: все потоки связи проходят через координатора. Подагенты имеют изолированный контекст — никогда не наследуют историю родительского агента автоматически. Инструмент задачи: координатор должен иметь "Task" в allowedTools. Всегда явно передавайте полный контекст в запросе подагента. Хуки: используйте хуки (а не запросы) для финансовых операций, соответствия требованиям, любых требований 100%-гарантии. PostToolUse для нормализации, PreToolUse для принудительного исполнения. Управление сессиями: --resume для именованных сессий, fork_session для параллельного сравнения подходов. Декомпозиция задач: фиксированные конвейеры для предсказуемых задач, динамическая декомпозиция для открытого исследования.

Раздел 2: Ключевые понятия проектирования инструментов

Описание инструментов является основным механизмом выбора. Минимальные описания приводят к неправильной маршрутизации. Что следует включать в описание: что делает и возвращает инструмент, форматы входных данных и примеры, граничные случаи, когда использовать аналогичные альтернативы. Структурированные ошибки из MCP: включают errorCategory, isRetryable, message, attempted_query, partial_results. Общее сообщение "Операция не удалась" предотвращает восстановление. tool_choice: • "auto" — модель принимает решение (по умолчанию) • "any" — должен вызывать инструмент (гарантированный структурированный вывод) • {"type": "tool", "name": "..."} — принудительное использование конкретного инструмента. Принцип наименьших привилегий: ограничить набор инструментов каждого субагента тем, что актуально для его роли.

Раздел 3: Ключевые понятия Кодекса Клода

Иерархия CLAUDE.md: пользователь → проект → каталог. Если у нового члена команды отсутствуют инструкции, они относятся к уровню пользователя, а не к уровню проекта. Импорт @path: модульная конфигурация, предотвращение дублирования. .claude/rules/ с путями: шаблоны glob для условной загрузки. Используйте для соглашений, охватывающих множество каталогов. Метатеги навыков: • context: fork — изолировать подробный вывод • allowed-tools — ограничить доступ к инструментам • argument-hint — запросить необходимые параметры. Режим планирования: большие изменения, несколько подходов, архитектурные решения, незнакомый код. CI/CD: всегда используйте флаг -p. Используйте --output-format json + --json-schema для структурированного вывода.

Раздел 4: Ключевые понятия оперативного проектирования

Краткое описание > текстовые инструкции для согласованного вывода. Обоснование неоднозначных сценариев, а не просто понятные примеры. Четкие критерии: точно определите, что следует пометить, а что игнорировать. Уровни серьезности с примерами кода. JSON Schema + tool_use: исключает синтаксические ошибки, а не семантические. Используйте поля, допускающие значение NULL, значения перечислений "unclear" и "other". Повторная попытка с обратной связью: включите исходный документ + предыдущее извлечение + конкретную ошибку. Повторная попытка не помогает, если информация отсутствует. Пакетный API: экономия 50%, задержка до 24 часов, отсутствие многоэтапных вызовов инструментов. Только для неблокирующих рабочих нагрузок. Многопроходная проверка: сначала проверка каждого файла, затем интеграционная проверка. Предотвращает рассеивание внимания.

Область 5: Ключевые понятия управления контекстом

Потеря в середине: размещайте ключевые результаты в НАЧАЛЕ агрегированного ввода. Не полагайтесь на информацию в середине контекста. Блок фактов дела: извлекайте суммы, даты, идентификаторы в постоянный структурированный блок вне истории суммирования. Числа сохраняются после суммирования. Инструмент обрезки выходных данных: хук PostToolUse для сохранения только 5 релевантных полей из более чем 40 возвращенных. Сохраняет контекстные токены. Распространение ошибок: возвращайте структурированный контекст (тип, запрос, частичные значения, альтернативы). Различайте сбой доступа от пустого результата. Аннотации покрытия в синтезе. Эскалация: явный запрос → немедленная эскалация. Пробел в политике → эскалация. Анализ настроения и уверенность в себе являются ненадежными триггерами. Множественные совпадения: всегда запрашивайте дополнительные идентификаторы, никогда не пытайтесь угадать.
